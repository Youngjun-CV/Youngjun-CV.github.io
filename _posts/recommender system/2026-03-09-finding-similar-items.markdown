## 🤔 왜 "유사도"를 수학적으로 정의해야 할까?

우리는 일상적으로 "이 영화랑 저 영화랑 비슷해"라고 말합니다. 하지만 컴퓨터에게 "비슷한 거 가져와"라고 하면 이해하지 못하죠. '비슷하다'라는 느낌은 매우 **주관적**이기 때문입니다.

추천 시스템의 핵심은 이 **주관적인 유사성을 수학적 지표로 객관화**하는 것입니다. 이를 통해 우리는 수만 개의 아이템 중 가장 관련성 높은 것을 "계산"해서 찾아낼 수 있게 됩니다. 이는 단순히 추천을 넘어, 데이터 사이의 숨겨진 관계를 찾아내는 과정이기도 합니다.

---

## 🔎 유사도 측정의 두가지 관점

수학적 모델을 선택하기 전, 우리는 **어떤 데이터를 기준으로 유사함을 정의할 것인가**를 먼저 결정해야 합니다. 이는 크게 두 가지 관점으로 나뉩니다.

### **(1) 아이템 특징 (Item Feature) 관점**
아이템이 가진 **고유한 속성**이 비슷한 경우를 말합니다. 유저의 행동과 상관없이 아이템 자체의 정보로 유사도를 판단합니다.
* **데이터 예시:** 영화의 장르, 감독, 배우, 제조사, 키워드 등
* **활용:** "이 영화는 저 영화와 같은 'SF' 장르이므로 비슷하다"고 정의할 때 사용합니다.

### **(2) 상호작용 데이터 (Interaction Data) 관점**
아이템 자체의 속성보다는 **사용자의 행동 패턴**이 비슷한 경우를 말합니다. 실제 유저들이 어떻게 반응했는지가 기준이 됩니다.
* **데이터 예시:** 클릭/구매/시청 기록, '좋아요/싫어요', 별점 기록 등
* **활용:** "두 영화를 시청한 유저 군집이 80% 일치하므로 두 영화는 비슷하다"고 정의할 때 사용합니다.

## 📐 데이터 타입별 유사도 측정 방법론

관점을 정했다면, 이제 데이터의 성격에 맞는 수학적 도구를 선택할 차례입니다.

### **(1) 자카드 유사도 (Jaccard Similarity): 집합의 언어**
데이터의 구체적인 수치보다 **'포함되었는가?'** 라는 이진(**Binary**) 정보가 중요할 때 사용합니다.

$$J(A, B) = \frac{|A \cap B|}{|A \cup B|} = \frac{|A \cap B|}{|A| + |B| - |A \cap B|}$$

* **공식 설명:** 두 집합의 합집합 크기 대비 교집합 크기의 비율을 계산합니다.
* **적용 예시 (영화 장르 분석):**
   * 영화 A: {스릴러, 액션, SF} / 영화 B: {액션, 코미디, 로맨스}
   * 교집합: {액션} (1개) / 합집합: {스릴러, 액션, SF, 코미디, 로맨스} (5개)
   * 결과: $1 / 5 =$ **0.2 (20%)**

### **(2) 코사인 유사도 (Cosine Similarity): 방향의 일치**
단순한 클릭 여부를 넘어 유저의 **긍정/부정 의사**가 담긴 데이터(좋아요/싫어요 등)를 비교할 때 최적입니다.

$$cos(\theta) = \frac{x \cdot y}{\|x\| \|y\|} = \frac{\sum_{i=1}^{n} x_i y_i}{\sqrt{\sum_{i=1}^{n} x_i^2} \sqrt{\sum_{i=1}^{n} y_i^2}}$$

* **공식 설명:** 분자는 벡터의 내적으로 일치도를, 분모는 정규화(**Normalization**)를 담당합니다.
* **적용 예시 (좋아요 +1 / 싫어요 -1):**
   * 유저 A의 평가: [영화1: 좋아요(+1), 영화2: 싫어요(-1), 영화3: 미평가(0)] $\rightarrow$ $\vec{x} = [1, -1, 0]$
   * 유저 B의 평가: [영화1: 좋아요(+1), 영화2: 좋아요(+1), 영화3: 좋아요(+1)] $\rightarrow$ $\vec{y} = [1, 1, 1]$
   * **분자(내적)**: $(1 \times 1) + (-1 \times 1) + (0 \times 1) = 0$
   * **분모(크기)**: $\sqrt{1^2 + (-1)^2 + 0^2} \times \sqrt{1^2 + 1^2 + 1^2} = \sqrt{2} \times \sqrt{3}$
   * **최종 유사도**: $0 / \sqrt{6} =$ **0** (두 사람의 취향은 아무런 상관관계가 없음)

### **(3) 피어슨 상관계수 (Pearson Correlation Coefficient): 편향 제거**
별점 같은 수치형 데이터에서 각 사용자의 **평균 별점**을 뺀 뒤 계산합니다. 깐깐한 사용자와 관대한 사용자의 주관적 차이를 보정해 주는 강력한 도구입니다.  

$$\rho_{x,y} = \frac{\sum (x_i - \bar{x})(y_i - \bar{y})}{\sqrt{\sum (x_i - \bar{x})^2} \sqrt{\sum (y_i - \bar{y})^2}}$$

* **공식 설명:** 모든 평점에서 유저의 **평균 평점**을 빼서 '평균 별점 = 0'이 되도록 보정(Mean Centering)합니다.
* **적용 예시 (평점 테러/후한 평가 보정):**
   * 유저 A의 별: [영화1: 2점, 영화2: 3점, 영화3: 4점] $\rightarrow$ 평균: 3점
   * **보정 후 벡터**: 각 점수에서 평균(3)을 뺌 $\rightarrow$ $\vec{a} = [-1, 0, 1]$
   * **보정의 의미**: 모든 영화에 낮은 점수를 주는 유저와 높은 점수를 주는 유저가 있더라도, 본인의 평균보다 높게 줬는지 낮게 줬는지를 기준으로 취향을 객관화합니다.
   * **최종 단계**: 이렇게 보정된 벡터들로 코사인 유사도를 계산하여 최종 상관계수를 산출합니다.

## 👆 데이터 형태에 따른 알고리즘 선택 가이드
데이터의 성격에 맞지 않는 알고리즘을 선택하는 것은 잘못된 렌즈로 세상을 보는 것과 같습니다. 상황별 최적의 선택 기준을 정리해봅시다.
* **Case 1: 단순 클릭/구매 이력만 있을 때**
   * 적합한 유사도: **Jaccard Similarity**
   * 선택 이유: 유저가 아이템을 선택했는지 여부만 기록된 **이진 데이터** (Binary Data) 상황에서는 집합 연산이 가장 직관적이고 빠릅니다.
* **Case 2: 좋아요/싫어요 피드백이 있을 때**
   * 적합한 유사도: **Cosine Similarity**
   * 선택 이유: 긍정(+1)과 부정(-1)의 방향성을 벡터로 표현하여 취향의 일치도를 계산할 수 있습니다.
* **Case 3: $1 \sim 5$ 점 사이의 구체적인 별점 데이터가 있을 때**
   * 적합한 유사도: **Pearson Correlation**
   * 선택 이유: 유저마다의 평점 기준(점수 편향)을 제거하고 순수한 선호도를 추출해야 하기 때문입니다.

## 🚨 우리가 놓치기 쉬운 '유사도'의 함정
유사도 알고리즘은 수학적으로 명쾌하지만, 현실 데이터에 적용할 때는 예상치 못한 **왜곡**이 발생할 수 있습니다. 유사도 알고리즘을 사용하기 전에 반드시 경계해야 할 함정입니다.

### (1) 결측치 (Missing Value) 0점 처리의 오류
추천 시스템 데이터셋에서 가장 흔한 실수는 평가하지 않은 항목(Empty)을 단순히 숫자 $0$으로 채우는 것입니다. 계산을 위해 빈칸을 채우는 것이 당연해 보이지만, 이는 수학적으로 매우 위험한 데이터 왜곡을 초례합니다.
* **구체적 사례:**
   * **서비스 환경:** $1$점 (최악) 부터 $5$점 (최고) 까지 줄 수 있는 영화 리뷰 사이트.
   * **유저 A:** SF 영화를 좋아해서 '인터스텔라'에 **5점**을 주었고, '로맨스 영화'는 아예 관심이 없어 평점을 남기지 않았습니다.
* **문제 발생:**
   * 컴퓨터는 이 유저의 데이터를 $[5, 0]$ (인터스텔라, 로맨스)으로 인식합니다.
   * 여기서 $0$ 은 $1$ 점보다 낮은 수치입니다. 즉, 수식 내에서 $0$ 은 '관심 없음'이 아니라 **'1점보다 더 싫어하는 극도의 혐오'** 로 계산에 반영됩니다.
* **결과:**
   * 시스템은 유저 A가 로맨스 장르를 '증오'한다고 오해하여, 관련 영화를 추천 목록에서 영원히 배제하거나 유사도 계산 시 큰 마이너스 요인으로 작동시킵니다.
* **해결책 (Pearson Correlation의 핵심):**
   * 유저의 평균 평점(예: 3.5점)을 구해 각 점수에서 뺍니다 **(Mean Centering)**.
   * 이렇게 하면 평가하지 않은 항목은 자연스럽게 제외되거나, 보정 후 $0$ 이 **'딱 평균적인 선호도'**를 뜻하게 되어 데이터 왜곡을 방지할 수 있습니다.
 
### (2) 자카드 유사도와 '헤비 유저(Heavy User)'의 역설
자카드 유사도는 두 집합이 얼마나 겹치는지만 볼 뿐, 각 유저가 가진 데이터의 **'전체 비중'**은 고려하지 않습니다.
* **구체적 사례:**
   * **유저 A (라이트 유저):** 영화를 딱 2편 보았고, 그중 하나가 '기생충'입니다. (기생충이 취향의 50% 차지)
   * **유저 B (헤비 유저):** 영화를 500편 보았고, 그중 하나가 '기생충'입니다. (기생충이 취향의 0.2% 차지)
* **문제 발생:**
   * 두 유저 모두 '기생충'이라는 공통 분모를 가졌으므로 자카드 유사도의 분자는 동일하게 $1$ 입니다. 유저 B는 거의 모든 영화를 다 보았기 때문에, 어떤 라이트 유저와 비교해도 항상 교집합이 발생합니다.
* **결과:**
   * 시스템은 유저 B를 '모든 사람과 취향이 비슷한 사람'으로 오해하게 됩니다. 이로 인해 유저 B가 본 방대한 양의 영화들이 다른 유저들에게 무분별하게 추천되어 개인화 성능이 저하됩니다.

### (3) 인기 편향 (Popularity Bias)과 필터 버블
인기 편향은 데이터 내에 압도적으로 많이 소비된 아이템이 존재할 때, 유사도 계산이 '취향'이 아닌 **'유명세'** 에 휘둘리게 되는 현상입니다.
* **구체적 사례:** 전국민이 다 본 메가 히트작 **'범죄도시'**가 있다고 가정해 봅시다.
   * **유저 X:** 평소 인디 영화만 즐기지만 '범죄도시'를 보았습니다.
   * **유저 Y:** 평소 공포 영화만 즐기지만 '범죄도시'를 보았습니다.
* **문제 발생:**
   * 두 유저는 취향이 극단적으로 다르지만, '범죄도시'라는 공통 기록 하나만으로 유사도 점수가 매우 높게 측정됩니다
* **결과:**
   * 시스템은 이 두 사람을 비슷한 집단으로 묶어버립니다. 인디 영화 팬에게 공포 영화를 추천하는 오류를 범하게 되고, 결과적으로 시스템은 모험을 하지 않고 누구에게나 무난한 **인기 아이템만 반복 추천** 하게 됩니다.

## 🫡 끝내며: 추천 시스템 설계의 본질
완벽한 유사도 알고리즘은 없습니다. 데이터의 형태가 이진 기록인지, 혹은 주관적인 별점인지에 따라 정답은 매번 달라지기 때문입니다.

단순히 수식을 계산하는 능력을 넘어, **평균 중심화(Mean Centering)** 와 같은 보정 과정을 거쳐 데이터의 왜곡을 줄여나가는 것, 그것이 추천 시스템을 설계할 때 우리 끊임없이 고민해야 할 본질입니다.
