## ☺️ 들어가며
최근 딥러닝 기술이 급격히 발전함에 따라 모델의 구조가 복잡해지고 파라미터 수가 폭발적으로 증가하며 모델이 점차 무거워지고 있습니다. 하지만 우리의 실제 목표는 고성능 GPU 서버뿐만 아니라, 스마트폰이나 임베디드 기기처럼 자원이 제한된 환경에서도 모델이 원활하게 동작하도록 만드는 것이며 이를 위해 모델 경량화는 선택이 아닌 필수입니다.

성공적인 경량화를 위해서는 무작정 모델을 줄이기보다, 현재 모델의 연산량과 메모리 사용량을 객관적으로 측정하여 최적화의 기준점을 세우는 과정이 선행되어야 합니다. 본격적인 경량화 기법을 다루기에 앞서, 이번 포스팅에서는 모델의 상태를 정확하게 판단할 수 있는 다양한 효율성 지표(Efficiency Metrics)에 대해 먼저 자세히 알아보겠습니다.

---

## 🔩 효율성 지표의 개요 (Efficiency Metrics)
모델의 효율성은 크게 **연산 복잡도** 와 **메모리 복잡도** 라는 두 가지 측면에서 평가됩니다. 단순히 "모델이 빠르다"라고 말하기보다, 어떤 자원을 얼마나 사용하고 있는지를 정량적으로 파악하는 것이 중요합니다.

### (1) 핵심평가 지표

* **Memory 관련 지표 (모델의 무게)**
  * **파라미터 수 (#parameters):** 모델에 포함된 학습 가능한 가중치의 총개수입니다.
  * **모델 크기 (model size):** 가중치들이 차지하는 실제 저장 공간 (주로 MB, GB 단위)입니다.
  * **활성화 함수의 총량 및 피크치 (total/peak #activations):** 추론 과정 중 각 레이어에서 생성되는 중간 데이터(Feature Map)의 양입니다. 이는 추론 시 필요한 런타임 메모리 병목의 주원인이 됩니다. 

* **Computation 관련 지표 (모델의 속도)**
  * **MAC (Multiply-Accumulate):** 딥러닝의 핵심인 "곱셈 후 더하기" 연산의 횟수입니다.
  * **FLOP/FLOPS:** FLOP은 모델의 총 부동 소숫점 연산 횟수를 의미하며, FLOPS는 초당 처리 가능한 연산 성능을 뜻합니다.
  * **OP/OPS:** 정수형 연산을 포함하여 일반화된 연산량과 그 초당 처리 능력을 나타냅니다.

### (2) 복잡도 최적화 전략
우리의 최종 목표는 더 작고 (Smaller), 더 빠르고 (Faster), 더 친환경적인 (Greener) 모델을 구현하는 것입니다. 이를 위해 하드웨어와 소프트웨어를 아우르는 최적화 전략이 필요합니다.

* **연산 복잡도 (Computation Complexity) 최적화**
  * 연산 시간(Latency)을 줄이기 위해서는 기본적으로 프로세서 코어 자체의 퍼포먼스가 높아야 합니다.
  * 고성능 코어는 초당 더 많은 연산을 수행하여 복잡한 수식 처리를 가속화합니다.

* **메모리 복잡도 (Memory Complexity) 최적화**
  * 딥러닝 연산에서 가장 큰 병목은 연산 자체가 아니라 데이터 이동에서 발생합니다.
  * 프로세서 외부 (Off-chip) 메모리인 DRAM과 데이터를 주고 받을 때 발생하는 Latency를 줄이는 것이 핵심입니다.
  * 이를 극복하기 위해 대역폭이 매우 넓은 HBM (High Bandwidth Memory) 기술을 사용하거나, 모델 자체를 압축하여 메모리 접근 횟수를 최소화해야 합니다.

## 🌿 주요 성능 지표 상세
모델의 효율성을 평가할 때, 우리는 '얼마나 빠른가'를 두 가지 관점에서 바라봐야 합니다. 하나는 개별 작업의 속도 (Latency)이고, 다른 하나는 전체적인 처리 능력 (Throughput) 입니다.

### (1) Latency (지연시간) vs Throughput (처리량)
* **Latency:** 특정 작업 하나를 완료하는 데 걸리는 시간(Delay)입니다. **CPU** 는 범용적이고 코어 수는 적지만 개별 연산 속도가 빨라, 실시간 응답이 중요한 모델의 Latency 최적화에 유리합니다.

* **Throughput:** 단위 시간당 처리되는 데이터 총량입니다. **GPU** 는 수만 개의 코어를 통해 대량의 데이터를 병렬 처리하므로, 배치(Batch) 단위의 대규모 데이터를 처리하는 Throughput 최적화에 특화되어 있습니다.

  * **Throughput 향상을 위한 프로세서 설계 방식**
    1. **Single Processor:** 단일 프로세서로 순차 처리
    2. **Parallel Processor:** 여러 프로세서를 평행하게 배치하여 동시에 처리
    3. **Pipelined Processor:** 단계별로 연산을 나누어 파이프라인화하여 처리
 
### (2) Latency Estimation
실제 디바이스에 모델을 올리기 전, 이론적으로 성능을 예측해 볼 수 있습니다. 연산 시간과 메모리 접근 시간 중 더 오래 걸리는 쪽이 전체 속도를 지배하게 됩니다. 

$$Latency \approx max(T_{computation}, T_{memory})$$

* **$T_{computation}$:** (네트워크 모델 총 연산 수) / (프로세서의 초당 연산 가능 개수) 
* **$T_{memory}$:** $T_{Data Movement Of Weights} + T_{Data Movement Of Activations}$
* **$T_{Data Movement Of Weights}$:** (모델 사이즈) / (메모리 대역폭, Bandwidth)
* **$T_{Data Movement Of Activations}$:** (Input Activation + Output Activation) / (메모리 대역폭)

여기서 중요한 점은 메모리 시간($T_{memory}$) 입니다. 실제 딥러닝 연산에서는 연산 자체보다 데이터를 메모리에서 코어로 가져오는 시간이 병목(Memory Wall)이 되는 경우가 많습니다.

### (3) 에너지 소비 (Energy Consumption)
에너지 효율 관점에서도 데이터 이동은 매우 치명적입니다. 데이터 처리 자체보다 데이터 이동 (Data movement)에 훨씬 많은 에너지가 소모됩니다.

* 32 bit DRAM 접근: 640 pJ 
* 32 bit float ADD 연산: 0.9 pJ 

DRAM (Off-chip)에 한 번 접근하는 것이 연산 한 번보다 **약 700배** 이상의 에너지를 더 사용합니다.

### (4) 경량화를 하는 이유
위에서 확인한 것 처럼, 속도를 높이고 에너지를 절약하려면 데이터를 외부 메모리 (Off-chip, DRAM)가 아닌 프로세서 내부 메모리(On-chip, SRAM) 내부에 저장하고 코어와 주고받는 것이 가장 효율적입니다. 

우리가 앞으로 배울 Pruning 이나 Quantization 같은 모델 압축 기술은 단순히 모델 크기를 줄이는 것이 목적이 아닙니다. 모델을 압축하여 가중치와 활성화 값의 크기를 줄임으로써, 값비싼 DRAM 접근을 최소화하고 데이터를 최대한 On-chip 메모리에 상주시켜 Latency와 에너지 효율을 동시에 잡기 위한 전략입니다.

## 📏 모델 사양 계산법 (Model Specification)
학습된 모델이 실제 디바이스에서 얼마만큼의 공간을 차지하고, 연산 시 어느 정도의 부하를 주는지 수치화하는 방법을 알아보겠습니다.

### (1) 파라미터 수 (# Parameters)
파라미터는 모델의 저장 공간 (Weight) 을 결정합니다. 각 레이어에서 우리가 학습시켜야 하는 가중치들의 총합입니다.

* **Linear Layer**
  * $C_{i} \times C_{o}$
  * 입력 채널($C_{i}$)의 모든 노드가 출력 채널($C_{o}$)의 모든 노드와 연결됩니다.

* **Convolution Layer**
  * $C_{i} \times C_{o} \times k_{w} \times k_{h}$
  * 필터(커널) 하나가 입력 채널 전체를 보며 이동합니다. 따라서 $C_{i} \times k_{w} \times k_{h}$ 크기의 커널이 출력 채널 개수($C_{o}$)만큼 존재하게 됩니다.

* **Grouped Conv**
  * $C_{i} \times C_{o} \times k_{w} \times k_{h} / g$
  * 입력/출력 채널을 여러 그룹($g$)으로 나누어 독립적으로 연산하므로 전체 파라미터 수는 $g$ 로 나눈 값과 같습니다.
  
* **Depthwise Conv**
  * $C_{o} \times k_{w} \times k_{h}$
  * 각 입력 채널마다 하나의 커널만 대응하므로 파라미터가 획기적으로 줄어듭니다.

AlexNet 의 경우 총 파라미터 수는 약 61M 이며, FP32 기준으로 모델 사이즈는 약 244 MB ($61M \times 4$ Bytes)가 됩니다.

### (2) 활성화 값 (Activations)
추론 시 입력 데이터가 레이어를 통과하며 생성되는 중간 데이터(Feature Map)의 크기입니다. 이는 실시간 메모리 점유율을 결정하는 핵심 요소입니다.

* **Total Activation:** 모델 전체 레이어에서 생성되는 모든 활성화 값의 합입니다. (AlexNet: 약 932,264) 

* **Peak Activation:** 특정 시점에 메모리에 가장 많이 올라가는 데이터의 양입니다. 보통 **입력 활성화 값 + 출력 활성화 값** 이 가장 큰 구간을 기준으로 삼습니다. (AlexNet: 약 440,928)

### (3) 연산량 (MAC, FLOP, OP)
* MAC: $a \leftarrow a + b \cdot c$ 연산 횟수입니다.
* FLOP: 1 MAC = 2 FLOP (곱셈 1 + 덧셈 1)으로 계산합니다.
* AlexNet 연산량: 약 724M MACs 로, 이를 FLOP 으로 환산하면 약 1.4 GFLOPs 가 됩니다.

## ✌️ 마치며
지금까지 인공지능 모델의 효율성을 평가하는 지표들을 살펴보았습니다. 단순히 파라미터가 적다고 해서 빠른 모델은 아닙니다. Latency Estimation 수식에서 보았듯이, 연산량뿐만 아니라 메모리 대역폭과 데이터 이동 시간을 종합적으로 고려해야 합니다.
