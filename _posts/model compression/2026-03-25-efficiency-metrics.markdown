## Introduction
최근 딥러닝 모델이 발전함에 따라 구조가 복잡해지고, 파라미터 수 또한 빠르게 증가하면서 모델의 규모가 점점 커지고 있다. 하지만 우리의 목표는 고성능 GPU 서버뿐만 아니라 스마트폰이나 임베디드 기기와 같은 자원이 제한된 환경에서도 모델을 효율적으로 동작시키는 것이다. 따라서 모델 경량화는 선택이 아닌 필수적인 과정이다.

효율적인 경량화를 위해서는 단순히 모델 크기를 줄이기보다, 현재 모델의 연산량과 메모리 사용량을 객관적으로 측정하고 기준을 설정하는 과정이 필요하다. 본격적인 경량화 기법을 다루기 전에, 이번 글에서는 모델의 상태를 정량적으로 평가할 수 있는 다양한 **효율성 지표(Efficiency Metrics)** 를 정리해보겠다.

---

## ✏️ 1. 모델의 정적 사양 (Model Static Specifications)
모델의 정적 사양은 크게 **연산 복잡도**와 **메모리 복잡도**라는 두 가지 측면에서 평가된다. 단순히 "모델이 빠르다"라고 말하기보다, 어떤 자원을 얼마나 사용하고 있는지 정량적으로 파악하는 것이 핵심이다. 각 지표가 구체적으로 무엇을 의미하는지 살펴보자.

###  1.1 Memory 관련 지표 (모델의 무게)
하드웨어에서 모델이 차지하는 물리적인 공간과 실행 시 필요한 메모리 점유율을 나타낸다.

  * **파라미터 수 (#parameters):** 모델에 포함된 학습 가능한 가중치(Weight)의 총개수다. 모델의 표현력을 결정하는 기준이 된다.
  
  * **모델 크기 (Model size):** 가중치들이 차지하는 실제 저장 공간 (주로 MB, GB 단위) 이다. 파라미터 수에 데이터 타입의 비트 수를 곱하여 구한다. (예: FP32라면 파라미터 수 $\times$ 4 Bytes)
  
  * **활성화 값의 총량 및 피크치 (total/peak #activations):** 추론 과정 중 각 레이어에서 생성되는 중간 데이터(Feature Map)의 크기다. 이 값들을 Off-chip 메모리(DRAM) 에서 가져와야 하기 때문에, 런타임 메모리 병목의 주원인이 된다.

### 🟩 Computation 관련 지표 (모델의 속도)
모델이 추론을 완료하기 위해 수행해야 하는 수학적 연산의 총량을 나타낸다.

  * **MAC (Multiply-Accumulate):** 딥러닝 연산의 기본 단위인 "곱셈 후 더하기($a \leftarrow a + b \times c$)"의 횟수다. 행렬 연산은 결국 이 MAC의 반복이기 때문에 가장 핵심적인 지표로 쓰인다.
  
  * **FLOP/FLOPS:** **FLOP** 은 부동 소수점 연산 횟수(Floating Point Operations)를, **FLOPS** 는 초당 처리 능력을 뜻한다. 보통 1 MAC을 2 FLOP(곱셈 1번 + 덧셈 1번) 으로 계산한다.
  
  * **OP/OPS:** 부동 소수점뿐만 아니라 정수형(Integer) 연산까지 모두 포함한 포괄적인 연산량과 그 초당 처리 능력을 의미한다.

## 🦾 모델의 동적 성능 (Model Dynamic Performance)
앞서 살펴본 정적 사양들이 실제 하드웨어에서 돌아갈 때 어떤 성능을 내는지 확인해야 한다. 모델의 효율성을 평가할 때 '얼마나 빠른가'는 두 가지 관점에서 바라봐야 한다. 하나는 개별 작업의 속도인 Latency 이고, 다른 하나는 전체적인 처리 능력인 Throughput 이다. 여기에 더해 실제 환경에서 중요한 에너지 소비량까지 포함해 알아보자.

### 🟩 Latency (지연시간) vs Throughput (처리량)
모델의 성능은 단순히 "빠르다"는 말 한마디로 정의되지 않는다. 서비스의 목적에 따라 중요하게 작용하는 지표가 다르기 때문이다.

#### ◼️ Latency (지연시간)
* **정의:** 특정 작업 하나를 완료하는 데 걸리는 시간(Delay)이다.
* **하드웨어 관점 (CPU):** 코어 수는 적지만 개별 코어의 연산 속도가 매우 빠르고 범용적이다. 그래서 하나하나의 작업을 빠르게 처리하여 낮은 Latency 를 달성하는 데 유리하다. 자율주행이나 실시간 번역처럼 즉각적인 응답이 중요한 서비스에서 핵심 지표가 된다.

#### ◼️ Throughput (처리량)
* **정의:** 단위 시간당 처리되는 데이터 총량이다.
* **하드웨어 관점 (GPU):** 개별 코어의 처리 능력은 CPU보다 낮을 수 있지만, 수만 개의 코어를 활용해 데이터를 병렬로 처리한다. 한 번에 많은 데이터를 밀어 넣는 배치(Batch) 처리에 최적화되어 있어 높은 Throughput 을 내기에 적합하다. 대규모 서버에서 사용자들의 요청을 한꺼번에 처리할 때 중요하다.

#### ◼️ 성능 향상을 위한 하드웨어 설계 방식
단순히 클럭 속도를 높이는 것 외에도, 하드웨어는 구조적으로 처리 효율을 높이기 위해 다양한 방식을 사용한다.

* **Single Processor (단일 프로세서):** 하나의 프로세서가 데이터를 순차적으로 처리하는 방식이다. 구조가 단순하지만, 한 번에 하나의 연산만 수행할 수 있어 대량의 데이터를 처리하기에는 한계가 있다.

* **Parallel Processor (병렬 프로세서):** 여러 개의 프로세서를 평행하게 배치하여 여러 데이터를 동시에(Simultaneously) 처리하는 방식이다. GPU가 대표적인 예시로, 전체 연산량을 여러 코어에 분산시켜 Throughput을 획기적으로 높인다.

* **Pipelined Processor (파이프라인 프로세서):** 연산 단계를 여러 스테이지로 세분화하여 공장 라인처럼 끊임없이 데이터가 흐르도록 만드는 방식이다. 앞선 데이터의 특정 단계 연산이 끝나면, 쉬지 않고 다음 데이터의 연산을 시작함으로써 전체적인 처리 효율을 극대화한다.

 
### 🟩 Latency Estimation (지연 시간 예측)
모델을 실제 디바이스에 올리기 전, 이론적인 계산을 통해 모델의 성능을 미리 예측해 볼 수 있다. 딥러닝 태스크에서 Latency 는 다음과 같은 수식으로 정의되며, **연산 시간** 과 **메모리 접근 시간** 중 더 오래 걸리는 쪽이 전체 속도를 결정하게 된다.

$$Latency \approx \max(T_{computation}, T_{memory})$$

#### ◼️ 연산 시간 ($T_{computation}$)
모델이 수행해야 할 순수 산술 연산에 걸리는 시간이다.

* **계산식:** (네트워크 모델 총 연산 수) / (프로세서의 초당 연산 가능 개수)
* 프로세서의 연산 성능(FLOPS)이 높을수록 이 수치는 낮아진다.

#### ◼️ 메모리 시간 ($T_{memory}$)
데이터를 메모리에서 코어로 가져오는 데 걸리는 시간이다. 가중치 시간 ($T_{weights}$)과 활성화 값 시간 ($T_{activations}$) 을 합하여 구한다.

* **$T_{Weights}$:** (모델 사이즈) / (메모리 대역폭, Bandwidth)
* **$T_{Activations}$:** (Input + Output Activation 크기) / (메모리 대역폭)
* 메모리 대역폭이 넓을수록 데이터를 한꺼번에 많이 옮길 수 있어 시간이 단축된다.

#### ◼️ 메모리 시간($T_{memory}$)이 중요한 이유
위 수식에서 알 수 있듯이, 아무리 고성능 프로세서를 사용해서 $T_{computation}$ 을 0에 가깝게 줄여도 $T_{memory}$ 가 크면 전체 Latency는 줄어들지 않는다. 실제 딥러닝 연산에서는 연산 자체보다 데이터를 메모리(DRAM)에서 코어로 가져오는 시간이 훨씬 오래 걸리는 경우가 많다. 이를 **Memory Wall** 현상이라고 부른다.

코어는 이미 계산을 끝내고 다음 데이터를 기다리는데, 메모리에서 데이터를 보내주는 속도가 느려 코어가 노는 상황이 발생하는 것이다. 결국 경량화와 가속을 위해서는 연산량뿐만 아니라 메모리 접근을 최소화해야한다.

### 🟩 에너지 소비 (Energy Consumption)
에너지 효율 관점에서도 데이터 이동은 매우 치명적이다. 사실 딥러닝 모델에서 전력을 가장 많이 잡아먹는 주범은 데이터 처리(연산) 자체가 아니라, 데이터를 옮기는 데이터 이동 과정에 있다.

* **32 bit DRAM 접근:** 640 pJ 
* **32 bit float ADD 연산:** 0.9 pJ 

수치를 보면 알겠지만, 외부 메모리인 **DRAM (Off-chip)** 에 한 번 접근하는 것이 연산 한 번을 수행하는 것보다 약 **700배** 이상의 에너지를 더 사용한다. 즉, 계산을 수백 번 하는 것보다 데이터를 한 번 읽어오는 게 하드웨어 입장에서는 훨씬 큰 부담인 셈이다.

### 🟩 경량화를 하는 진짜 이유
위에서 확인했듯이, 속도를 높이고 에너지를 절약하려면 데이터를 외부 메모리(Off-chip, DRAM)가 아닌 프로세서 내부 메모리(On-chip, SRAM)에 저장하고 코어와 주고받는 것이 가장 효율적이다. 하지만 모델이 너무 크면 이 좁고 비싼 내부 메모리에 다 들어가지 못하고 결국 느리고 비싼 DRAM을 계속 왔다 갔다 해야 한다.

우리가 앞으로 배울 **Pruning** 이나 **Quantization** 같은 모델 압축 기술은 단순히 모델 크기를 줄여서 저장 공간을 아끼는 것이 목적이 아니다. 모델을 압축해서 가중치와 활성화 값의 크기를 줄임으로써, 값비싼 DRAM 접근을 최소화하고 데이터를 최대한 On-chip 메모리에 상주시키는 것이 핵심이다. 이를 통해 Latency와 에너지 효율을 동시에 잡으려는 전략이다.

## 💾 메모리 사양 계산법
지금까지 각 지표가 무엇을 의미하는지 개념을 잡았다면, 이제는 실제 모델의 구조를 보고 수치를 직접 계산해낼 차례다. 모델이 디바이스에서 얼마만큼의 물리적 공간을 차지하고, 실행 시 메모리 자원을 얼마나 점유하는지 수학적으로 계산하는 법을 알아보자.

### 🟩 파라미터 수 (# Parameters)
파라미터는 모델의 가중치(Weight)가 차지하는 저장 공간을 결정한다. 레이어별로 우리가 학습시켜야 할 가중치가 총 몇 개인지를 계산하면 된다.

#### ◼️ Linear Layer (Fully Connected Layer)
  * $C_{i} \times C_{o}$
  * 입력 채널($C_{i}$)의 모든 노드가 출력 채널($C_{o}$)의 모든 노드와 1:1로 연결되므로 두 값을 곱하면 된다.

#### ◼️ Convolution Layer
  * $C_{i} \times C_{o} \times k_{w} \times k_{h}$
  * 커널 하나당 파라미터 수($C_{i} \times k_{w} \times k_{h}$)에 커널의 개수인 출력 채널 수($C_{o}$)를 곱한 값이다.

#### ◼️ Grouped Conv
  * $C_{i} \times C_{o} \times k_{w} \times k_{h} / g$
  * 입력과 출력 채널을 $g$ 개의 그룹으로 나누어 독립적으로 연산하기 때문에, 일반 Conv 연산량에서 그룹 수 $g$ 만큼 나눈 것과 같다.
  
#### ◼️ Depthwise Conv
  * $C_{o} \times k_{w} \times k_{h}$
  * 각 입력 채널마다 단 하나의 커널(채널 수 1)만 대응한다. 즉, $C_{i} = 1$ 인 Conv가 $C_{o}$ 개 있는 꼴이라 파라미터가 획기적으로 줄어든다.

### 🟩 모델 사이즈 (Model Size)
모델 사이즈는 앞서 구한 파라미터 수에 가중치 하나를 표현하는 **Bit-width** 를 곱해서 구한다. 같은 모델이라도 어떤 정밀도(Precision)로 저장하느냐에 따라 무게가 달라진다.

* **계산식:** 파라미터 수 $\times$ 데이터 타입 크기 (Bytes)
* **AlexNet 예시 (약 61M Parameters):**
   * **FP32 (4 Bytes):** $61M \times 4 \approx 244 \text{ MB}$
   * **INT8 (1 Byte):** $61M \times 1 \approx 61 \text{ MB}$

### 🟩 활성화 값 (Activations)
추론 시 레이어를 통과하며 생성되는 중간 데이터(Feature Map)의 크기다. 가중치는 한 번 로드하면 고정이지만, 활성화 값은 입력 데이터가 들어올 때마다 실시간으로 생성되므로 메모리 점유율을 결정하는 핵심 요소가 된다.

* **Total Activation:** 모델의 모든 레이어에서 생성되는 활성화 값의 총합이다.

* **Peak Activation:** 특정 시점에 메모리에 가장 많이 올라가는 데이터 양이다. 하드웨어는 현재 레이어를 계산하기 위해 **[입력 활성화 값 + 출력 활성화 값]** 을 동시에 들고 있어야 하므로, 이 합이 가장 큰 구간이 병목 지점이 된다.

## 📐 연산 사양 계산법
앞서 모델이 디바이스에서 얼마만큼의 물리적 공간을 차지하는지 알아보았으니, 이번에는 모델이 실제로 실행될 때 얼마나 많은 계산을 수행하는지 알아보자. 모델의 연산량은 추론 속도(Latency)와 직결되므로, 실제 하드웨어 단위인 MAC과 FLOP을 기준으로 정밀하게 계산하는 법을 익히는 것이 중요하다.

### 🟩 MAC (Multiply-Accumulate)
딥러닝 연산의 가장 기본 단위다. $a \leftarrow a + (b \times c)$ 처럼 곱셈 하나와 덧셈 하나가 합쳐진 한 묶음을 1 MAC이라고 부른다. 실제 학습 환경에서는 한 번에 여러 데이터를 처리하는 배치 사이즈 ($B$) 를 고려해야 하므로, 모든 수식에 $B$ 가 곱해진다고 이해하면 된다.

#### ◼️ Linear Layer
* $B \times C_{i} \times C_{o}$
* 입력 채널과 출력 채널이 만나는 모든 지점에서 MAC 연산이 발생한다.

#### ◼️ Convolution Layer
* $B \times C_{i} \times k_{w} \times k_{h} \times C_{o} \times W_{o} \times H_{o}$
* 파라미터 수($C_{i} \times k_{w} \times k_{h} \times C_{o}$)에 출력 피처맵의 크기($W_{o} \times H_{o}$)를 곱한 값이다. 커널이 이미지 전체를 훑으며 반복 연산하기 때문에 연산량이 매우 크다.

#### ◼️ Grouped Conv
* $(B \times C_{i} \times C_{o} \times k_{w} \times k_{h} \times W_{o} \times H_{o}) / g$
* 그룹 수($g$)만큼 연산이 나누어지므로, 일반 Conv 연산량을 $g$ 로 나눈 것과 같다.

#### ◼️ Depthwise Conv
* $B \times C_{o} \times k_{w} \times k_{h} \times W_{o} \times H_{o}$
* 입력 채널 하나당 커널 하나가 대응($C_{i}=1$)하므로 연산량이 획기적으로 줄어든다.

### 🟩 FLOP (Floating Point Operations)
부동 소수점 연산 횟수를 의미한다. 하드웨어 스펙 시트에서 가장 많이 보이는 단위이기도 하다.
* **1 MAC = 2 FLOPs:** 1 MAC 안에는 곱셈 1번과 덧셈 1번이 들어있으므로, FLOP으로 환산할 때는 MAC 수에 2를 곱해준다.

### 🟩 OP (Operations)
부동 소수점(Float)뿐만 아니라 정수형(Integer), 논리 연산 등 모든 종류의 산술 연산을 포괄하는 단위다. 연산의 종류와 상관없이 하드웨어가 처리해야 하는 총 업무량이라고 이해하면 쉽다.

## ✌️ 마치며
지금까지 인공지능 모델의 효율성을 평가하는 다양한 지표들을 살펴보았다. 이번 포스팅을 통해 알 수 있듯이, 단순히 파라미터가 적다고 해서 무조건 빠른 모델은 아니다. Latency Estimation 수식에서 확인했듯, 연산량뿐만 아니라 메모리 대역폭과 데이터 이동 시간을 종합적으로 고려해야 진짜 '가벼운' 모델을 만들 수 있다.

결국 Efficiency Metrics 를 정확히 파악하는 것은 스마트폰이나 임베디드 기기처럼 한정된 자원 내에서 최적의 퍼포먼스를 내는 모델을 설계하기 위한 첫걸음이다. 내 모델이 어디서 병목(Bottle-neck)이 발생하는지 알아야 제대로 된 처방을 내릴 수 있기 때문이다.

이 지표들을 바탕으로 다음 포스팅에서는 본격적인 경량화 기법의 첫 번째 단계인 푸르닝 (Pruning) 에 대해 알아보겠다.
