## Introduction
최근 딥러닝 모델이 발전함에 따라 구조가 복잡해지고, 파라미터 수 또한 빠르게 증가하면서 모델의 규모가 점점 커지고 있다. 하지만 우리의 목표는 고성능 GPU 서버뿐만 아니라 스마트폰이나 임베디드 기기와 같은 자원이 제한된 환경에서도 모델을 효율적으로 동작시키는 것이다. 따라서 모델 경량화는 선택이 아닌 필수적인 과정이다.

효율적인 경량화를 위해서는 단순히 모델 크기를 줄이기보다, 현재 모델의 연산량과 메모리 사용량을 객관적으로 측정하고 기준을 설정하는 과정이 필요하다. 본격적인 경량화 기법을 다루기 전에, 이번 글에서는 모델의 상태를 정량적으로 평가할 수 있는 다양한 **효율성 지표(Efficiency Metrics)** 를 정리해보겠다.

---

## ✏️ 1. 모델의 정적 사양 (Model Static Specifications)
모델의 정적 사양은 크게 **연산 복잡도**와 **메모리 복잡도**라는 두 가지 측면에서 평가된다. 단순히 "모델이 빠르다"라고 말하기보다, 어떤 자원을 얼마나 사용하고 있는지 정량적으로 파악하는 것이 핵심이다. 각 지표가 구체적으로 무엇을 의미하는지 살펴보자.

###  1.1 Memory 관련 지표 (모델의 무게)
하드웨어에서 모델이 차지하는 물리적인 공간과 실행 시 필요한 메모리 점유율을 나타낸다.

* **파라미터 수 (#parameters):** 모델에 포함된 학습 가능한 가중치(weight)의 총개수다. 모델의 표현력을 결정하는 기준이 된다.
  
  * **모델 크기 (Model size):** 가중치들이 차지하는 실제 저장 공간(주로 MB, GB 단위)이다. 파라미터 수에 데이터 타입의 비트 수를 곱하여 구한다. (예: FP32라면 파라미터 수 $\times$ 4 Bytes)
  
  * **활성화 값의 총량 및 피크치 (total/peak #activations):** 추론 과정에서 각 레이어가 생성하는 중간 데이터(feature map)의 크기다. 이 값들은 off-chip 메모리(DRAM)에서 가져와야 하므로, 런타임 메모리 병목의 주요 원인이 된다.

### 1.2 Computation 관련 지표 (모델의 속도)
모델이 추론을 완료하기 위해 수행해야 하는 수학적 연산의 총량을 나타낸다.

* **MAC (Multiply-Accumulate):** 딥러닝 연산의 기본 단위인 "곱셈 후 덧셈($a \leftarrow a + b \times c$)"의 횟수다. 행렬 연산은 결국 MAC의 반복으로 이루어지므로 핵심적인 지표로 사용된다.

* **FLOP/FLOPS:** **FLOP**은 부동소수점 연산 횟수(Floating Point Operations)를, **FLOPS**는 초당 처리 능력을 의미한다. 보통 1 MAC은 2 FLOP(곱셈 1번 + 덧셈 1번)으로 계산한다.
  
* **OP/OPS:** 부동소수점뿐만 아니라 정수형(integer) 연산까지 포함한 포괄적인 연산량과 그 초당 처리 능력을 의미한다.

## 🦾 2. 모델의 동적 성능 (Model Dynamic Performance)
앞서 살펴본 정적 사양들이 실제 하드웨어에서 돌아갈 때 어떤 성능을 내는지 확인해야 한다. 모델의 효율성을 평가할 때 '얼마나 빠른가'는 두 가지 관점에서 바라봐야 한다. 하나는 개별 작업의 속도인 Latency 이고, 다른 하나는 전체적인 처리 능력인 Throughput 이다. 여기에 더해 실제 환경에서 중요한 에너지 소비량까지 포함해 알아보자.

### 2.1 Latency (지연시간) vs Throughput (처리량)
모델의 성능은 단순히 “빠르다”는 말로 정의되지 않는다. 서비스의 목적에 따라 중요한 지표가 달라지기 때문이다.

#### ◼️ Latency (지연시간)
* **정의:** 하나의 작업을 완료하는 데 걸리는 시간이다. 
* **하드웨어 관점 (CPU):** 코어 수는 적지만 개별 코어의 성능이 높고 범용적이다. 따라서 개별 작업을 빠르게 처리하여 낮은 latency를 달성하는 데 유리하다. 자율주행이나 실시간 번역처럼 즉각적인 응답이 필요한 서비스에서 중요한 지표다.

#### ◼️ Throughput (처리량)
* **정의:** 단위 시간당 처리되는 데이터의 양이다.
* **하드웨어 관점 (GPU):** 개별 코어 성능은 CPU보다 낮을 수 있지만, 수많은 코어를 활용한 병렬 처리를 통해 높은 처리량을 달성한다. 배치(batch) 처리에 최적화되어 있어 대규모 요청을 동시에 처리하는 환경에서 중요하다.

#### ◼️ 성능 향상을 위한 하드웨어 설계 방식
<img src="/images/ModelCompression/effi_mat_1.jpg">
하드웨어는 단순히 클럭 속도를 높이는 것 외에도, 구조적인 방법으로 처리 효율을 향상시킨다.

* **Single Processor (단일 프로세서):** 하나의 프로세서가 데이터를 순차적으로 처리하는 방식이다. 구조는 단순하지만, 동시에 처리할 수 있는 연산이 제한된다.

* **Parallel Processor (병렬 프로세서):** 여러 프로세서를 병렬로 배치하여 데이터를 동시에 처리하는 방식이다. GPU가 대표적인 예시로, 연산을 여러 코어에 분산시켜 throughput을 높인다.

* **Pipelined Processor (파이프라인 프로세서):** 연산 과정을 여러 단계로 나누어, 공정 라인처럼 연속적으로 데이터를 처리하는 방식이다. 각 단계가 동시에 동작하여 전체 처리 효율을 향상시킨다.
 
### 2.2 Latency Estimation
모델을 실제 디바이스에 올리기 전, 이론적인 계산을 통해 모델의 성능을 미리 예측해 볼 수 있다. 딥러닝 태스크에서 Latency 는 다음과 같은 수식으로 정의되며, **연산 시간** 과 **메모리 접근 시간** 중 더 오래 걸리는 쪽이 전체 속도를 결정하게 된다.

$$Latency \approx \max(T_{computation}, T_{memory})$$

#### ◼️ 연산 시간 ($T_{computation}$)
모델이 수행하는 순수 연산에 소요되는 시간이다.

* **계산식:** (총 연산 수) / (프로세서의 초당 연산 처리량)
* **특징:** 프로세서의 연산 성능(FLOPS)이 높을수록 해당 시간은 감소한다.

#### ◼️ 메모리 시간 ($T_{memory}$)
데이터를 메모리에서 코어로 전달하는 데 걸리는 시간이다. 가중치 시간과 활성화 값 시간을 합산하여 계산한다.

* **가중치 시간 ($T_{Weights}$):** (모델 크기) / (메모리 대역폭)
* **활성화 값 시간 ($T_{Activations}$):** (입력 + 출력 activation 크기) / (메모리 대역폭)
* **특징:** 메모리 대역폭이 클수록 데이터 전송 시간이 감소한다.

#### ◼️ 메모리 시간($T_{memory}$)이 중요한 이유
위 식에서 알 수 있듯이, $T_{computation}$을 아무리 줄이더라도 $T_{memory}$가 크다면 전체 latency는 개선되지 않는다. 실제 딥러닝에서는 연산보다 메모리 접근 시간이 더 큰 비중을 차지하는 경우가 많으며, 이를 **memory wall** 문제라고 한다.

코어는 계산을 마치고 다음 데이터를 기다리지만, 메모리에서 데이터를 전달하는 속도가 느려 유휴 상태가 발생한다. 따라서 효율적인 모델 설계를 위해서는 연산량 감소뿐만 아니라 메모리 접근 최소화가 필수적이다.

### 2.3 에너지 소비 (Energy Consumption)
<img src="/images/ModelCompression/effi_mat_2.jpg" style="width:70%;">
에너지 관점에서도 데이터 이동 비용은 매우 크다. 딥러닝 시스템에서 전력 소모의 주요 원인은 연산 자체보다 데이터 이동에 있다.

외부 메모리인 DRAM(off-chip)에 접근하는 비용은 단순 연산보다 수백 배 이상 크다. 즉, 계산을 여러 번 수행하는 것보다 데이터를 한 번 이동시키는 것이 더 큰 에너지 비용을 유발한다.

### 2.4 경량화를 하는 이유
효율적인 연산을 위해서는 데이터를 외부 메모리(DRAM)가 아닌 내부 메모리(SRAM, on-chip)에 최대한 유지하는 것이 중요하다. 하지만 모델이 클수록 모든 데이터를 on-chip 메모리에 저장할 수 없고, 결국 반복적인 DRAM 접근이 발생한다.

**Pruning**과 **Quantization**과 같은 모델 압축 기법은 단순히 모델 크기를 줄이는 것이 아니라, 데이터 이동을 최소화하는 데 목적이 있다. 가중치와 activation 크기를 줄여 on-chip 활용도를 높이고, 결과적으로 latency와 에너지 효율을 동시에 개선한다.

## 💾 3. 메모리 사양 계산법
지금까지 각 지표의 의미를 정리했다면, 이제 모델 구조를 기반으로 실제 수치를 계산할 차례다. 모델이 디바이스에서 차지하는 저장 공간과 실행 시 필요한 메모리 사용량을 어떻게 계산하는지 살펴보자.

### 3.1 파라미터 수 (# Parameters)
파라미터 수는 모델의 가중치(weight)가 차지하는 저장 공간을 결정한다. 레이어별로 학습해야 할 가중치의 개수를 계산하면 된다.

#### ◼️ Linear Layer (Fully Connected Layer)
* **수식:** 입력 채널($C_{i}$)의 모든 노드가 출력 채널($C_{o}$)의 모든 노드와 연결되므로 두 값을 곱해 계산한다.

$$C_{i} \times C_{o}$$

#### ◼️ Convolution Layer
* **수식 :** 하나의 커널이 가지는 파라미터 수($C_{i} \times k_{w} \times k_{h}$)에 커널의 개수인 출력 채널 수($C_{o}$)를 곱한 값이다.

$$C_{i} \times C_{o} \times k_{w} \times k_{h}$$

#### ◼️ Grouped Conv
* **수식 :** 입력과 출력 채널을 $g$개의 그룹으로 나누어 독립적으로 연산하므로, 일반 convolution 대비 파라미터 수가 $g$배 감소한다.

$$C_{i} \times C_{o} \times k_{w} \times k_{h} / g$$
  
#### ◼️ Depthwise Conv
* **수식 :** 각 입력 채널마다 하나의 커널만 적용되는 구조로, 파라미터 수가 크게 감소한다.

$$C_{o} \times k_{w} \times k_{h}$$
  

### 3.2 모델 사이즈 (Model Size)
모델 사이즈는 파라미터 수에 가중치 하나를 표현하는 **bit-width**를 곱하여 계산한다. 동일한 모델이라도 사용하는 정밀도에 따라 저장 크기가 달라진다.

* **수식** : 파라미터 수에 데이터 타입 크기를 구하여 계산한다.

$$\text{Model Size} = |\text{Parameters}| \cdot \text{Bit Width}$$

### 3.3 활성화 값 (Activations)
활성화 값은 추론 과정에서 레이어를 통과하며 생성되는 중간 데이터(feature map)의 크기를 의미한다. 가중치는 고정되어 한 번 로드되지만, 활성화 값은 입력마다 새롭게 생성되므로 메모리 사용량에 큰 영향을 미친다.

* **Total Activation :** 모든 레이어에서 생성되는 활성화 값의 총합이다.

* **Peak Activation :** 특정 시점에서 메모리에 동시에 존재하는 최대 데이터 양이다. 일반적으로 **입력 activation** 과 **출력 activation** 의 합이 가장 큰 구간이 병목이 된다.

## 📐 4. 연산 사양 계산법
앞서 모델이 디바이스에서 차지하는 저장 공간을 살펴봤다면, 이제 모델이 실제로 실행될 때 필요한 연산량을 계산해보자. 연산량은 추론 속도(latency)와 직접적으로 연결되므로, MAC과 FLOP 기준으로 정량적으로 이해하는 것이 중요하다.

### 4.1 MAC (Multiply-Accumulate)
딥러닝 연산의 기본 단위로, $a \leftarrow a + (b \times c)$와 같이 곱셈과 덧셈이 결합된 연산을 의미한다. 실제 환경에서는 배치 크기($B$)를 고려하므로, 전체 연산량에 $B$가 곱해진다.

#### ◼️ Linear Layer
* **수식 :** 입력 채널과 출력 채널이 연결되는 모든 위치에서 MAC 연산이 수행된다.

$$B \times C_{i} \times C_{o}$$

#### ◼️ Convolution Layer
* **수식 :** 파라미터 수에 출력 feature map 크기를 곱한 형태로, 커널이 전체 입력을 순회하며 연산을 수행하기 때문에 연산량이 크다.

$$B \times C_{i} \times k_{w} \times k_{h} \times C_{o} \times W_{o} \times H_{o}$$

#### ◼️ Grouped Conv
* **수식 :** 채널을 $g$개의 그룹으로 나누어 처리하므로, 전체 연산량이 $g$배 감소한다.

$$(B \times C_{i} \times C_{o} \times k_{w} \times k_{h} \times W_{o} \times H_{o}) / g$$

#### ◼️ Depthwise Conv
* **수식 :** 각 입력 채널마다 하나의 커널만 적용되므로 연산량이 크게 감소한다.

$$B \times C_{o} \times k_{w} \times k_{h} \times W_{o} \times H_{o}$$


### 4.2 FLOP (Floating Point Operations)
부동소수점 연산 횟수를 의미하며, 하드웨어 성능을 표현할 때 자주 사용되는 단위다.
* **1 MAC = 2 FLOPs :** 1 MAC은 곱셈 1번과 덧셈 1번으로 구성되므로 FLOP 기준에서는 2배로 환산된다.

### 4.3 OP (Operations)
부동소수점뿐만 아니라 정수형(integer), 논리 연산 등 모든 연산을 포함하는 개념이다. 연산의 종류와 관계없이 하드웨어가 처리해야 하는 전체 작업량을 나타낸다.

## Conclusion
지금까지 인공지능 모델의 효율성을 평가하는 다양한 지표를 살펴보았다. 단순히 파라미터 수가 적다고 해서 반드시 빠른 모델이라고 볼 수는 없다. latency estimation에서 확인했듯, 연산량뿐만 아니라 메모리 대역폭과 데이터 이동 시간까지 함께 고려해야 실제로 효율적인 모델을 설계할 수 있다.

결국 efficiency metrics를 정확히 이해하는 것은 제한된 자원 환경에서 최적의 성능을 내기 위한 출발점이다. 모델에서 어떤 부분이 병목(bottleneck)을 유발하는지 파악해야 적절한 최적화 전략을 적용할 수 있다.

다음 글에서는 이러한 지표를 바탕으로, 모델 경량화의 핵심 기법 중 하나인 **Pruning**을 다뤄보겠다.
